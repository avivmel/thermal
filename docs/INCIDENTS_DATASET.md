# Incidents Dataset Documentation

## Overview

The incidents dataset extracts goal-oriented thermal episodes from raw Ecobee NetCDF data. Unlike the flat `thermal_dataset.csv` which contains arbitrary timesteps, this dataset organizes data into meaningful episodes where the HVAC system is pursuing a thermal goal.

| Attribute | Value |
|-----------|-------|
| Source | Raw Ecobee NetCDF files |
| Location | `data/incidents/` |
| Format | Parquet (compressed) |
| Generated by | `scripts/prepare_incidents.py` |

---

## Output Files

```
data/incidents/
├── heating.parquet    # Heating episodes (~20M rows)
├── cooling.parquet    # Cooling episodes (~7M rows)
├── drift.parquet      # Drift episodes (~74M rows)
├── setpoint.parquet   # Setpoint change episodes (~10M rows)
└── summary.parquet    # One row per episode (for quick analysis)
```

Each file is split by incident type to allow loading only what's needed for specific experiments.

---

## Incident Types

### 1. Heating Episode

| Attribute | Description |
|-----------|-------------|
| **Start** | Indoor temp drops below heat setpoint |
| **End** | Indoor temp >= heat setpoint (recovery) OR mode changes to cooling |
| **is_active** | True when outdoor < indoor (HVAC fighting cold outside) |

Heating episodes capture when the home is below the desired temperature and the HVAC system is working to warm it up.

### 2. Cooling Episode

| Attribute | Description |
|-----------|-------------|
| **Start** | Indoor temp rises above cool setpoint |
| **End** | Indoor temp <= cool setpoint (recovery) OR mode changes to heating |
| **is_active** | True when outdoor > indoor (HVAC fighting hot outside) |

Cooling episodes capture when the home is above the desired temperature and the AC is working to cool it down.

### 3. Drift Episode

| Attribute | Description |
|-----------|-------------|
| **Start** | Transition from heating/cooling zone to deadband (HVAC goal achieved) |
| **End** | Indoor temp exits deadband OR setpoint change OR 4-hour timeout |
| **is_active** | Always False (no HVAC intervention) |

Drift episodes capture the passive thermal behavior when HVAC is off. Useful for modeling how quickly homes lose/gain heat.

### 4. Setpoint Change Episode

| Attribute | Description |
|-----------|-------------|
| **Start** | Heat setpoint increases OR cool setpoint decreases by >0.5°F |
| **End** | Temperature reaches new setpoint OR 2-hour timeout |
| **is_active** | Based on thermal gradient at each timestep |

Setpoint change episodes capture the system's response to user/schedule interventions. Critical for MPC since we need to predict "what happens if I change the setpoint?"

## Schema

### Incident Metadata Columns

| Column | Type | Description |
|--------|------|-------------|
| `incident_id` | string | Unique ID: `{home_id}_{incident_type}_{start_timestamp}` |
| `home_id` | string | Anonymized home identifier |
| `timestamp` | datetime | Timestamp of this row |
| `state` | string | US state code (CA, TX, IL, NY) |
| `split` | string | train/val/test (matches thermal_dataset.csv) |
| `incident_type` | string | heating, cooling, drift, or setpoint_change |
| `is_active` | bool | True if HVAC fighting thermal gradient |
| `timestep_idx` | int | 0-indexed position within episode |
| `episode_duration` | int | Total timesteps in episode |
| `episode_start` | datetime | When episode began |
| `episode_end` | datetime | When episode ended |
| `start_indoor_temp` | float | Indoor temp at episode start (°F) |
| `start_outdoor_temp` | float | Outdoor temp at episode start (°F) |
| `target_setpoint` | float | Setpoint being pursued (°F) |
| `initial_gap` | float | Distance from setpoint at start (°F) |

### Preserved Original Columns

All 29 variables from raw NetCDF files are preserved:

**Temperatures (°F)**
- `Indoor_AverageTemperature` - Weighted average of all sensors
- `Outdoor_Temperature` - Outdoor air temperature
- `Thermostat_Temperature` - Temperature at thermostat
- `RemoteSensor1-5_Temperature` - Remote sensor temperatures

**Setpoints (°F)**
- `Indoor_HeatSetpoint` - Heating setpoint
- `Indoor_CoolSetpoint` - Cooling setpoint

**Runtime (seconds per 5-min interval, max 300)**
- `HeatingEquipmentStage1/2/3_RunTime` - Heating equipment runtime
- `CoolingEquipmentStage1/2_RunTime` - Cooling equipment runtime
- `HeatPumpsStage1/2_RunTime` - Heat pump runtime
- `Fan_RunTime` - Fan runtime

**Humidity (%)**
- `Indoor_Humidity` - Indoor relative humidity
- `Outdoor_Humidity` - Outdoor relative humidity

**Motion (binary)**
- `Thermostat_DetectedMotion` - Motion at thermostat
- `RemoteSensor1-5_DetectedMotion` - Motion at remote sensors

**Other**
- `Schedule` - Current schedule mode (Home, Away, Sleep, or empty)
- `Event` - Active event (Hold, custom events, or empty)
- `HVAC_Mode` - HVAC operating mode (-9999 = missing)

## Usage Examples

### Load Heating Episodes

```python
import pandas as pd

# Load all heating episodes
heating = pd.read_parquet("data/incidents/heating.parquet")

# Filter to training data only
train_heating = heating[heating["split"] == "train"]

# Get first timestep of each episode (for trajectory prediction)
first_steps = heating[heating["timestep_idx"] == 0]
```

### Load Summary for Analysis

```python
import pandas as pd

summary = pd.read_parquet("data/incidents/summary.parquet")

# Episode duration statistics by type
print(summary.groupby("incident_type")["episode_duration"].describe())

# Average initial gap by state
print(summary.groupby("state")["initial_gap"].mean())
```

### Extract Trajectories

```python
import pandas as pd

heating = pd.read_parquet("data/incidents/heating.parquet")

# Get a single episode trajectory
episode_id = heating["incident_id"].iloc[0]
trajectory = heating[heating["incident_id"] == episode_id].sort_values("timestep_idx")

# Plot temperature over time
import matplotlib.pyplot as plt
plt.plot(trajectory["timestep_idx"] * 5, trajectory["Indoor_AverageTemperature"])
plt.xlabel("Minutes since episode start")
plt.ylabel("Indoor Temperature (°F)")
plt.title("Heating Episode Trajectory")
```

### Filter by Active Episodes

```python
# Get only episodes where HVAC was fighting thermal gradient
active_heating = heating[heating["is_active"]]

# Get episodes that were mostly active (>80% of timesteps)
summary = pd.read_parquet("data/incidents/summary.parquet")
mostly_active = summary[summary["pct_active"] > 0.8]
```

## Design Notes

### Mode Classification (Setpoint-Based)

Episodes are defined by setpoint comparison, not runtime:

```python
if indoor_temp < heat_setpoint:
    mode = "heating"
elif indoor_temp > cool_setpoint:
    mode = "cooling"
else:
    mode = "deadband"
```

This is goal-oriented: we care about the thermal state relative to user preferences, not whether equipment is currently running.

### Active vs Passive

The `is_active` flag indicates whether HVAC is fighting the thermal gradient:

- **Active heating**: outdoor < indoor (cold outside makes heating harder)
- **Active cooling**: outdoor > indoor (hot outside makes cooling harder)
- **Passive**: thermal gradient assists HVAC goal

92% of heating is active, but only 36% of cooling is active (often the outdoor temp has cooled by evening when indoor is still warm).

### Cross-Month Episodes

Episodes that span month boundaries are split into separate episodes. This is acceptable since:
- Most episodes are short (median ~25 min for heating)
- Less than 1% of episodes span month boundaries
- Split episodes still provide valid training data

### Missing Data Handling

- Episodes are closed when missing data (0.0 for temps) is encountered
- New episode starts when valid data resumes
- This prevents episodes from spanning data gaps

## Generation

To regenerate the dataset:

```bash
# Full run (all 12 months, ~971 homes)
python scripts/prepare_incidents.py

# Test mode (January only, 5 homes)
python scripts/prepare_incidents.py --test

# Test with more homes
python scripts/prepare_incidents.py --test --n-homes 20
```

Estimated runtime for full dataset: ~2-3 hours (depends on disk I/O).
